Using Jira as workflow engine with Zapier/ Slack/ Gsuite / Symfony/ Canvas

Aug 6, 2017 | Tech

Recently I have a chance to manage a project which across a lot of different pieces of software. The goal and outcome is very interesting and I would like to share with others about it.

So the initial application which is built on symfony is about helping educators on boarding to different institutions and when it does, there will be a lot of manual work involve:

  • Create account in Gsuite admin(google business suite with gmail, calendars, drive..etc)
  • Create account in Slack for support and instant collaborative work
  • Create account in Canvas which is a community tools for academic

Also there is not enough visual ability to monitor all different cases happening in the application, well, we do have a status tag in the Symfony application itself to show to the operator but we want a better visual like a Trello board to display clearly who is currently waiting for interview, waiting for reference check or who has been rejected…etc

Lastly, most of the time the person who decided the workflow will not be the same guy who is writing the code. So in order to let the business and operation team have full visual and control on the work flow, we need a workflow engine which is able to lock the current process inside our Symfony application and the application can only update the case status according to the flow from the workflow engine.

So why Jira?
There is a lot of existing software solution in the market but none of them can fit into our requirement entirely. Since there will be some hard work involve in any solution, why don’t we choose something that we are familiar, have great API and a UI which “approved” from our business development team.

So how to connect them together?
I think most of the coders will either start creating a service inside our symfony application or just build a application to pass through the data in between. The funny part of this project is, one of the requirement is to allow some room for non-technical people to trace up which data has been sent and receive, status of each call and even let them modify what need to be passed in each API. In this case, we will need a tools which can provide a clean and easy API for all the API calls status and also a drag and drop tools for our team to modify the API data. And that’s why we end up using Zapier as our middle man.

I will share some codes in Part 2 which I have used in this project and the steps of how I setup those service particularly those mistakes that I have made during the process.
